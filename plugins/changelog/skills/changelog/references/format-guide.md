# CHANGELOG.md Format Guide

This reference contains the full formatting specification for CHANGELOG.md files generated by this skill. Changelogs are for **users, developers, and operators** — not raw commit logs or JIRA dumps.

## Template

```markdown
# Changelog

All notable changes to this project are documented in this file.
The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

## [1.0.0] – 2026-03-30

Initial stable release.

### Added

- Core functionality: describe main feature here.

### Changed

- Default behavior for X: explain briefly.

### Deprecated

- Feature that will be removed in v2.0: explain briefly.

### Removed

- Legacy flag or API that is no longer supported.

### Fixed

- Fix for crash when Y happens.

### Security

- Patch for vulnerability in Z (CVE-XXXX-XXXX if applicable).

[Unreleased]: https://github.com/user/project/compare/v1.0.0...HEAD
[1.0.0]: https://github.com/user/project/releases/tag/v1.0.0
```

## Overall Structure

- One section per **version** (e.g., `## [1.2.0] – 2026-03-30`), in **reverse chronological order** (newest on top).
- Each version entry lists the **release date** and then grouped changes under semantic categories.
- The file starts with `# Changelog` and a brief intro paragraph.
- An `[Unreleased]` section at the top collects changes not yet in a tagged release.
- Reference-style version comparison links at the bottom (no heading, just bare links).

## Change Categories

Use these headings in this order. Omit empty categories from the output:

| Category | What belongs here |
|----------|------------------|
| `Added` | New features, APIs, config options, commands |
| `Changed` | Modifications to existing behavior, parameters, defaults |
| `Deprecated` | Features that will be removed in a future release; mention the target major version if possible |
| `Removed` | Already-removed features or APIs; triggers a MAJOR semver bump |
| `Fixed` | Bug fixes; focus on user-visible impact, not internal stack traces |
| `Security` | Vulnerabilities, patches, or security-related config changes |

## Commit-to-Category Mapping

When analyzing git commits, classify them as follows:

| Commit signals | Category |
|---------------|----------|
| `feat:`, "add", "new", "introduce", "implement" | Added |
| `refactor:`, "update", "change", "rename", "migrate", "improve" | Changed |
| `deprecate:`, "deprecate", "sunset" | Deprecated |
| `remove:`, "drop", "delete", "strip" | Removed |
| `fix:`, "bug", "patch", "resolve", "correct", "repair" | Fixed |
| `security:`, "CVE", "vulnerability", "auth fix" | Security |

**Priority:** When a conventional commit prefix is present (e.g., `feat:`, `fix:`, `refactor:`), use the prefix to determine the category. Only fall back to keyword matching in the message body when no recognized prefix exists.

**Skip merge commits** — Merge commits (e.g., "Merge branch ...", "Merge pull request ...") duplicate the merged commits and should be skipped.

If a commit does not clearly fit any category (e.g., pure CI/CD, docs-only, or trivial chore), omit it unless it is user-facing.

## Style Rules

1. **For humans, not machines** — Changelogs exist for humans. Do not dump raw `git log` or noise from refactors.
2. **Short bullet points** — Synthesize commits into user-facing descriptions, not verbatim commit messages.
3. **Consistent past tense** — "Added X", "Fixed Y", "Removed Z".
4. **Capitalize first word** of each bullet.
5. **Be specific and user-focused** — Instead of "Fixed bug", write "Fixed crash when opening large files".
6. **Mention breaking implications** — If a change is breaking, say so explicitly.
7. **Link to issues/PRs** when helpful — e.g., `(#42)` or `([PR #15](url))`.
8. **Omit empty categories** — Only include category headings that have entries.
9. **Never rewrite or delete old entries** — If a correction is needed, document it in the current version's section.
10. **Easily linkable** — Versions and sections use Markdown headings for easy deep-linking.

## Workflow and Maintenance

- Keep an `[Unreleased]` section at the top and add changes there as work progresses.
- When cutting a new release, rename `## [Unreleased]` to `## [X.Y.Z] – YYYY-MM-DD`, then add a fresh `## [Unreleased]` above it.
- Breaking changes live under `Removed` or `Changed` and correspond to `MAJOR` semver bumps.
- New backwards-compatible features correspond to `MINOR` bumps.
- Bug fixes correspond to `PATCH` bumps.

## Version Detection

Versions are determined from git tags matching semver patterns:
- `v1.2.3`, `1.2.3`, `v1.0.0-beta.1`
- Fall back to chronological grouping if no tags exist.
- The `[Unreleased]` section captures commits after the latest tag.

## Version Links

At the bottom of the file (no heading, just bare reference-style links), maintain comparison links:

```markdown
[Unreleased]: https://github.com/user/project/compare/v1.2.0...HEAD
[1.2.0]: https://github.com/user/project/compare/v1.1.0...v1.2.0
[1.1.0]: https://github.com/user/project/compare/v1.0.0...v1.1.0
[1.0.0]: https://github.com/user/project/releases/tag/v1.0.0
```

Detect the remote URL from `git remote get-url origin` and construct links accordingly. Support both GitHub and GitLab URL formats.

## Common Mistakes to Avoid

### Copy-pasting commit logs
Raw `git log` or PR titles ("fix typo", "refactor foo") are noisy and not user-facing. Group multiple commits into one user-visible change and summarize the outcome, not the technical steps.

### Being too vague or generic
"Various improvements" or "Bug fixes and performance improvements" without specifics. Users cannot tell if the update matters to them. Be specific: "Fixed crash when uploading large files" or "Reduced page load time by up to 40%."

### Ignoring breaking changes
Not clearly calling out breaking changes or hiding them in a long list. Use `Removed` or `Changed` headings, describe the effect, and provide a migration path (e.g., "API v1 endpoints removed; use v2 instead").

### Inconsistent structure or tone
Mixing formats between versions (sometimes grouped by category, sometimes chronologically). Stick to the standard headings and keep bullets short and uniform.

### Information overload
Listing every tiny tweak, internal refactor, or doc fix that does not affect users. Only include **notable changes** — things users will see or feel. Put purely internal changes in internal docs or PR summaries.

### Too technical without context
Writing only for developers with jargon like "Refactored auth middleware for JWT validation." Explain the user-facing effect: "Login is now more reliable across different browsers."

### Skipping releases
Only documenting "big" releases and leaving minor/patch versions empty erodes trust. If there are no notable changes, say so explicitly: "No user-facing changes; internal fixes only."

### Batching changes
Letting many changes accumulate and dumping them all into one giant entry. Keep the `[Unreleased]` section updated incrementally, then split into version-specific entries at release time.

## Handling Breaking Changes

Breaking changes require **visibility**, **clarity**, and **migration guidance**.

### Make breaking changes highly visible

Surface them prominently. Prefix entries with a bold label:

```markdown
- **BREAKING:** Field `foo` is removed from the response payload.
```

Breaking changes correspond to a **MAJOR** semver bump and appear only in `Changed` or `Removed` sections, not `Added`.

### Describe the impact, not just the code change

Do not say "Removed legacy auth flow." Describe what users will experience:

```markdown
- **BREAKING:** Legacy API keys are no longer accepted; all clients must now use OAuth 2.0.
```

For libraries or APIs, mention which interfaces changed, which parameters are removed, and what behavior changed.

### Use deprecation before removal

First **deprecate** in one version (warn users), then **remove** in a later version:

```markdown
### Deprecated (will be removed in v2.0)
- The `old_serialize()` method is deprecated; use `serialize_v2()` instead.
```

This gives ecosystem users time to adapt.

### Provide migration instructions

For each breaking change, add a short migration path:

```markdown
- **BREAKING:** Switch from `client.connect()` to `client.connect_v2()` and pass `timeout: 30`.
- Update config file `config.yaml` to rename `old_key` to `new_key`.
```

For complex migrations, link to a migration guide or docs page.

### Be consistent in labeling

Use the inline `**BREAKING:**` prefix for all breaking changes. This is scannable and works with both manual and auto-generated changelogs.

### Never hide breaking changes

Do not bury breaking changes in a long list of minor tweaks under `Changed`. They must stand out — either through the `**BREAKING:**` prefix or by being listed first within their category section.
